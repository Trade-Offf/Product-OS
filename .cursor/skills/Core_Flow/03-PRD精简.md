# PRD 精简技能（可选环节）

## 使用场景

- 原始 PRD（如 `03_Output/PRD_*.md`）**结构完整但过重**，对小需求或局部改动来说阅读成本太高。
- 上下游同事只需要：
  - 这个需求 **是什么 & 为什么做**（背景 + 目标）
  - **对现状改了什么**（关键变更点）
  - **有哪些硬规则/数值约束**（方便开发/测试快速对齐）
  - **潜在风险或依赖**（避免遗漏）
- 不需要完整的 Code Truth、全量流程图、长表格等细节，只要一眼能看懂的"精简版 PRD"。

> 该技能是 **可选补充环节**，不会影响主流程的需求整理和正式 PRD 生成。

---

## 输入要求

- 必须已经存在一份完整 PRD，通常位于：
  - `03_Output/PRD_*.md`
- 不再直接读取 `01_Input/` 原始需求，而是 **以完整 PRD 为唯一输入源** 进行二次提炼。

> 严格遵守"不新增需求、不修改逻辑"原则，仅做 **信息压缩与重排**。

---

## 输出要求

- 建议输出到与原 PRD 同目录的轻量文件，例如：
  - `03_Output/PRD_*_light.md`
- 输出内容控制在 **1–2 页以内**（大致 300–800 行内文本，视实际情况裁剪），优先保证：
  - 信息 **密度高**
  - 章节结构 **简单稳定**
  - 一眼能看到 **关键变更 & 硬规则**

### 推荐文档结构

使用下列稳定结构生成精简版 PRD：

```markdown
# {项目名称} 精简版 PRD

## 1. 背景 & 目标（短版）
- 用 2–4 条要点，说明：
  - 业务背景（只保留与本次改动直接相关的部分）
  - 本次改动的核心目标（不展开讨论历史）

## 2. 本次改动范围
- 列出本次改动涉及的：
  - 端：Web / App / 后端 / 管理后台（只写有改动的）
  - 模块 / 页面：简单描述即可
  - 不在本次范围内的内容，可用一句话说明"不改"

## 3. 关键变更点
- 用列表列出 3–8 条最重要的变更点：
  - 每条最多 1–2 句话
  - 尽量"一条 = 一个可以开发/测试的点"

示例：
- 新增「AI参数推荐」按钮于合约网格下单区（Web/App），点击后自动填充价格区间、网格数、杠杆等字段。
- 中性场景推荐网格数固定为 8 格，多空场景固定为 6 格，复用现有网格数校验。

## 4. 数字 & 规则速查
- 将 **对开发/测试影响最大的数值和硬规则** 用表格列出来：

| 项目       | 值 / 规则描述          | 说明（可选） |
| ---------- | --------------------- | ------------ |
| 中性价格区间 | 当前市价上下各 6%       |              |
| 中性网格数   | 固定 8 格             |              |

## 5. 风险 / 依赖 / 待定项（如有）
- 从正式 PRD 的"遗留/待定事项"中，挑出与本次需求执行密切相关的 3–5 条，简短列出。
```

## 精简规则（如何从完整 PRD 抽取）

### 1️⃣ 背景 & 目标（取精华，不讲故事）

- 来源章节：
  - `1. 背景与目标` 中的：
    - `1.1 业务背景`
    - `1.2 核心目标`
- 处理方式：
  - 只保留 **与本次改动直接相关** 的背景信息。
  - 删除长段历史叙事、无关演进过程。
  - 将多段文字汇总为 2–4 条要点。

### 2️⃣ 本次改动范围（回答"改哪里"）

- 来源章节：
  - `3. 功能详细设计` 中按端/模块拆解的部分
  - `4. 变更对比` 中提到的变更维度
- 处理方式：
  - 明确本次涉及的端（Web/App/API/后台），**只列有变更的端**。
  - 简要列出相关页面/模块名称或场景，如"合约网格下单区（Web）"。
  - 如果在正式 PRD 中明确写了"不改"的部分，可以在此用一句话说明（例如："现有手动创建入口不调整"）。

### 3️⃣ 关键变更点（回答"改了什么"）

- 来源章节：
  - `3. 功能详细设计`
  - `4. 变更对比`
  - `5. 限制规则` 中与变更强相关的部分
- 处理方式：
  - 用 **列表形式** 总结 3–8 条关键变更。
  - 每条尽量对应一个可以独立开发/测试的点：
    - "一个字段新增/变更"
    - "一个流程分支的变化"
    - "一条重要校验规则的变更或新增"
  - 避免把多个不相干点揉成一句长话。

### 4️⃣ 数字 & 规则速查（给开发/测试的速查表）

- 来源章节：
  - `5. 限制规则`
  - 以及其他章节中 **出现明确数值/阈值/枚举限制** 的地方
- 处理方式：
  - 只收集"硬规则"：数值区间、固定枚举、最小/最大限制等。
  - 用单列表格展示，列数控制在 3–4 列（项目 / 值 / 来源或说明）。
  - 避免解释性长句，优先用简洁短语。

### 5️⃣ 风险 / 依赖 / 待定项（只保留关键的）

- 来源章节：
  - `7. 遗留/待定事项`
- 处理方式：
  - 从完整列表中挑选 **与当下版本实施强相关** 的 3–5 条。
  - 优先选择：
    - 会影响是否能按期上线的决策
    - 会影响技术方案/数据埋点的决策
    - 跨团队依赖或需要额外排期的事项
  - 用短句列出，无需展开详细背景。

## 执行步骤（给 AI 自己看的操作流程）

1. **确定目标 PRD 文件**
   - 如果用户在对话中已经明确指定（例如提到 `03_Output/PRD_ai_param_recommend_v1.0_20260310.md`），则使用该文件。
   - 如果未指定，先列出 `03_Output/` 下最新的 `PRD_*.md` 文件，让用户选择或默认使用最新一份。

2. **读取完整 PRD**
   - 使用 Read 工具读取选定的 PRD 文件全文。

3. **按章节抽取信息**
   - 从 `1. 背景与目标` 中抽取 **业务背景** 与 **核心目标**，压缩为 2–4 条要点。
   - 从 `3. 功能详细设计`、`4. 变更对比`、`5. 限制规则` 中识别 **本次改动范围** 与 **关键变更点**。
   - 从包含数值和限制的章节汇总 **数字 & 规则速查表**。
   - 从 `7. 遗留/待定事项` 中筛选出 3–5 条 **最相关** 的事项。

4. **生成精简版 PRD 内容**
   - 按"推荐文档结构"章节给出的模板拼装：
     - 标题使用原 PRD 的项目名称 +「精简版 PRD」后缀。
     - 各章节中严格使用 **列表 + 简短句子 + 小表格** 的形式。
   - 明确说明：这是基于某个正式 PRD 的精简版，并在开头标注原始 PRD 文件名。

5. **写入输出文件**
   - 文件命名建议：在原 PRD 文件名后追加 `_light` 后缀，例如：
     - 原始：`PRD_ai_param_recommend_v1.0_20260310.md`
     - 精简：`PRD_ai_param_recommend_v1.0_20260310_light.md`
   - 写入到 `03_Output/` 目录。

6. **向用户回报结果**
   - 告知用户：
     - 生成的精简 PRD 文件名与路径
     - 文档结构中包含了哪些关键部分（例：背景&目标 / 改动范围 / 关键变更点 / 数字&规则速查 / 风险&依赖）
   - 强调：精简版只做信息压缩，不新增/修改任何需求逻辑，如需细节请以原 PRD 为准。

---

## 自检清单

在认为精简完成之前，需确认：

- [ ] 精简文档只基于 **一个完整 PRD**，未直接引用 `01_Input/` 原始输入。
- [ ] 未添加任何原 PRD 中不存在的新功能、流程或规则。
- [ ] 背景&目标部分控制在 2–4 条要点内，没有长篇叙述。
- [ ] "关键变更点"是一个可供开发/测试执行的清单（3–8 条），而不是模糊描述。
- [ ] "数字&规则速查"只包含硬规则和关键信息，没有灌水解释。
- [ ] 风险/依赖/待定项不超过 5 条，且与当前版本执行密切相关。
- [ ] 输出文件命名清晰（推荐使用 `_light` 后缀），路径位于 `03_Output/`。
