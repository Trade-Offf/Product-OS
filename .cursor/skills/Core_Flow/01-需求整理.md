# Core Flow / 01-需求整理

> 本技能遵循 `.cursorrules` 中的全局约束（严格忠实、状态机、主流程优先）。

## 技能目标

把 `01_Input/` 中的原始输入整理为标准化的 `_WORK_SHEET.md`。

本技能只负责：读取输入 → 提取事实 → 结构化整理 → 产出工作台 → 标记待确认问题

本技能**不负责**：生成 PRD（正式或草稿）、补充未提供的功能细节、调用扩展技能

---

## 输入范围

### 必读
- `01_Input/*`（PDF / Markdown / 文本等）

### 条件必读
- 当 `01_Input/Context.md` 存在且非空时，必须读取并融合历史背景

### 可选补充
- 用户在对话中额外提供的说明（不得覆盖 `01_Input/` 中的既有事实，除非用户明确纠正）

### 排除内容
- 功能拆分表格（前后端分工、估时等）
- 排期甘特图、开发时间线、人员安排
- 图片内容（由产品在 PRD 阶段自行补充）

---

## 输出文件

- 主输出：项目根目录 `_WORK_SHEET.md`
- 禁止输出：`03_Output/` 下的任何文件

---

## 执行步骤

### 第一步：识别输入材料

检查 `01_Input/` 中有哪些文件。如果存在 `Context.md`，读取并记录"有 Context"。

### 第二步：抽取原始事实

从输入中提取：功能名称、涉及端/平台、页面/模块、操作路径、目标行为、展示规则、限制条件、依赖关系、异常情况、原文中已明确的边界。

只提取事实，不推导新规则，不补充 PM 未写明的流程。

### 第三步：分端拆解

强制拆分为：**[Web端]** / **[App端]** / **[管理后台]** / **[API/后端]**

每个端内按页面/URL 分组。如果原文未明确 URL 或页面名称，标记 `> ❓ 待确认：该功能所属页面`。不编造 URL 路径。

### 第四步：规则提取与异常捕获

抓取数字、限制、约束条件，归类到限制规则。识别兜底逻辑和异常处理，归类到异常流程。只记录原文提到的，不推断。

### 第五步：识别缺口与矛盾

- 无法确认的内容 → `> ❓ 待确认：xxx`
- 原文中存在冲突 → 文档末尾增加 `## ⚠️ 发现的问题`
- Context.md 与新需求矛盾 → `> ⚠️ 冲突：Context.md 说...，新需求说...`
- 每个待确认问题必须引用触发该问题的原文

### 第六步：代码引用标记

当需求涉及已有功能修改时（优化、改进、状态机变更等），标记需要扫描的代码：

```markdown
> 📦 需扫描代码：{关键词/文件名/功能名}
> 建议扫描：*{关键词}*.java / *{关键词}*.ts 等
```

### 第七步：生成标准化工作台

生成 `_WORK_SHEET.md`，必须包含以下结构：

```markdown
---
created_at: {时间}
last_updated: {时间}
status: pending | ready
pending_count: {数字}
answered_count: {数字}
---

# _WORK_SHEET

> 📊 **当前状态**: ... | 待回答问题: ...个 | 已回答: ...个

## Part 1: 待确认问题汇总
### Q1 [⏳ 待回答]
**来源**: 01-需求整理
**原文**: "[引用触发该问题的原文]"
**问题**: [问题描述]
**[回答]**:

## Part 2: 详细拆解报告
### 1. 核心目标
### 2. 分端拆解
### 3. 规则整理
### 4. 限制规则
### 5. 异常流程
### 6. 原文摘录

## Part 3: 代码扫描建议
| 功能点 | 建议扫描关键词 | 原因 |
| ------ | -------------- | ---- |
```

---

## 状态判定

- 存在待确认问题 → `status: pending`，`是否允许正式生成：否`
- 待确认问题为 0 → `status: ready`，`是否允许正式生成：是`

---

## 输出后要求

整理完成后，告知用户：
- 已生成/更新 `_WORK_SHEET.md`
- 当前状态（`pending` 或 `ready`）
- 若为 `pending`，列出待确认问题数量
- 明确说明：当前不能直接视为正式 PRD

---

## 一句话行为准则

"整理阶段只产出工作台，不产出正式 PRD；有疑问就挂起到待确认，不自行补洞。"
