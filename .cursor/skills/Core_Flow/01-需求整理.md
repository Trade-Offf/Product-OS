# Skill: Requirement Organizer（需求整理器）

## 📋 技能依赖声明

### 必须依赖

- ✅ `01_Input/` 目录存在
- ✅ `01_Input/` 中至少有一个需求文件（PDF / Markdown / 文本）

### 可选依赖

- 📚 `01_Input/Context.md`：Lark 历史业务背景（如存在则必须读取）

### 依赖检查失败时的提示

```
⚠️ 需求整理需要输入文件，请先将需求文档放入 01_Input/ 目录
支持格式：PDF、Markdown (.md)、文本文件
可选：将 Lark 历史背景粘贴到 01_Input/Context.md
```

---

## 🛡️ Safety Protocol（防幻觉协议）

- **Strict Scope**: 仅提取文档中显式提到的逻辑，不发明、不脑补、不优化。
- **Gap Flagging**: 遇到逻辑断层时，标注 `> ❓ 待确认：...`，严禁自动补全。
- **Question Context**: 每个待确认问题必须引用触发该问题的原文。
- **Source Tracking**: 每个功能点都标注来源（原文："xxx"）或 `(来源: Context.md)`。
- **File Output Only**: 结果必须写入 `_WORK_SHEET.md`，严禁在对话框直接输出分析全文。
- **Text Focus**: 专注提取文字逻辑，图片由产品在 PRD 阶段自行补充。
- **Exclude Dev Content**: 排除开发阶段的内容（功能拆分表格、排期安排、开发人员信息等）。
- **Context First**: 如果 `01_Input/Context.md` 存在，必须先读取并融合历史背景。

## 📝 处理规则

### 0. 上下文融合（Context.md）

**执行条件**：`01_Input/Context.md` 文件存在且非空

**处理步骤**：
1. 读取 `01_Input/Context.md` 全文
2. 提取以下信息（如有）：
   - 历史业务背景
   - 之前的相关决策
   - 遗留问题或技术债
   - 关键数据/指标
3. 在 `_WORK_SHEET.md` 的 Part 0 中输出背景摘要

**标注规则**：
- 来自 Context.md 的内容标注 `(来源: Context.md)`
- 如果 Context.md 与新需求存在矛盾，标注 `> ⚠️ 冲突：Context.md 说...，新需求说...`

### 1. 分端拆解

强制拆分为以下分类：

- **[Web端]**: PC 浏览器访问的功能
- **[App端]**: iOS/Android 原生应用功能
- **[管理后台]**: 运营/管理员使用的后台系统
- **[API/后端]**: 纯后端逻辑，无前端界面

### 2. 页面级别拆解

在每个端的分类下，按 **页面/URL** 进一步分组：

**拆解原则**：

- 同一个 URL/页面路径下的所有功能，归为一组
- 每个页面组包含：页面路径、页面功能列表
- 如果原文未明确提及 URL，按页面名称分组（如"个人中心页"、"订单列表页"）
- 如果连页面名称都没有，标记 `> ❓ 待确认：该功能所属页面`

**防脑补规则**：

- 如果原文未说明在哪个页面 → 标记 `❓ 待确认`
- 如果原文未提及 URL → 写页面名称，URL 留空或标记待确认
- 不要自己编造 URL 路径结构

### 3. 规则提取

抓取所有数字、限制、约束条件，归类到 **[限制规则]**。

- **原则**: 只记录原文明确提到的限制，不推断。

### 4. 异常捕获

识别兜底逻辑和异常处理，归类到 **[异常流程]**。

- **原则**: 如果原文未提及异常处理，标注 `> ❓ 待确认`。

### 5. 代码引用标记

当需求涉及**已有功能的修改**时，标记需要扫描的代码：

**触发条件**：
- 需求提到"优化"、"改进"、"修改"某个现有功能
- 需求引用了具体的参数、限制、规则（可能来自代码）
- 需求涉及状态机、流程变更

**标记格式**：
```markdown
> 📦 需扫描代码：{关键词/文件名/功能名}
> 建议扫描：*{关键词}*.java / *{关键词}*.ts 等
```

**示例**：
- 需求说"修改网格交易的止盈逻辑" → `> 📦 需扫描代码：网格交易、止盈、GridStrategy`
- 需求说"当前最小金额是 20U" → `> 📦 需扫描代码：最小金额、MIN_AMOUNT`

### 5. PDF 文档处理

**排除开发内容**（重要）：

- ❌ 功能拆分表格（前后端分工、估时等）
- ❌ 排期甘特图、开发时间线
- ❌ 开发人员投入安排、头像信息

这些是需求评审通过后，开发团队填写的内容，不属于需求阶段。

**图片处理**：

- 不处理图片内容，图片由产品在 PRD 阶段自行补充
- 专注提取文字描述的功能逻辑

## 📤 输出规则

**输出路径**: `_WORK_SHEET.md`（项目根目录，唯一文件）

### 文件结构

创建或覆盖根目录下的 `_WORK_SHEET.md` 文件，包含三部分：

**Part 0: 业务背景摘要**（如 Context.md 存在）

- 从 `01_Input/Context.md` 提取的历史背景
- 格式：

  ```markdown
  ## Part 0: 业务背景

  ### 历史沿革
  {从 Context.md 提取}

  ### 历史决策
  {从 Context.md 提取}

  ### 遗留问题
  {从 Context.md 提取，如无则写"无"}

  > 📄 完整背景见 01_Input/Context.md
  ```

**Part 1: 待确认问题汇总**

- 供 PM 直接填写
- 每个问题包含：来源、原文引用、问题描述、回答区域
- 问题格式：

  ```markdown
  ### Q{序号} [⏳ 待回答]

  **来源**: 01-需求整理
  **原文**: "[引用触发该问题的原文]"
  **问题**: [问题描述]
  **[回答]**:
  ```

**Part 2: 详细拆解报告**

- AI 生成的完整分析内容
- 包含：核心目标、分端拆解、限制规则、异常流程、依赖关系、原文摘录

**Part 3: 代码扫描建议**（如有标记）

- 汇总所有 `> 📦 需扫描代码` 标记
- 格式：

  ```markdown
  ## Part 3: 代码扫描建议

  生成 PRD 前，建议扫描以下代码以确保"现状逻辑"准确：

  | 功能点 | 建议扫描关键词 | 原因 |
  | ------ | -------------- | ---- |
  | {功能} | {关键词}       | {为什么需要扫描} |

  **使用方法**：在执行"生成文档"时，@ 引用相关代码文件
  ```

### 元数据头部

在文件顶部添加 YAML 格式的元数据：

```yaml
---
created_at: { 创建时间 }
last_updated: { 最后更新时间 }
status: pending
pending_count: { 待回答问题数量 }
answered_count: 0
---
```

在元数据后添加状态说明：

```markdown
> 📊 **当前状态**: 待确认 | 待回答问题: {数量}个 | 已回答: 0个
```
