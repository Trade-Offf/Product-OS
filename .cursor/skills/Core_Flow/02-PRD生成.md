# Core Flow / 02-PRD生成

> 本技能遵循 `.cursorrules` 中的全局约束（严格忠实、状态机、主流程优先）。

## 技能目标

基于 `_WORK_SHEET.md` 的状态，严格区分三种结果：拒绝生成 / 生成草稿 / 生成正式 PRD。

---

## 执行前检查

### 必读文件
1. `_WORK_SHEET.md`
2. `docs/templates/PRD_Template.md`（若存在）
3. `01_Input/Context.md`（若存在且非空）

### 条件必读
- 仅当用户在 prompt 中 `@代码文件` 时，读取相关代码文件

---

## 状态判定逻辑

### A. 拒绝正式生成

满足任一条件时拒绝：`_WORK_SHEET.md` 不存在 / 状态为 `pending` / 待确认 > 0 / `是否允许正式生成：否`

拒绝时告知用户当前状态、原因、可否改为草稿。

### B. 草稿生成

当用户明确要求草稿，或同意草稿方式输出时可执行。

- 路径：`03_Output/PRD_{功能名}_v{版本}_{日期}_draft.md`
- 标题明确标识「草稿」
- 保留待确认问题章节
- 生成后更新 `_WORK_SHEET.md` 状态为 `draft`

### C. 正式生成

同时满足：`_WORK_SHEET.md` 存在 / 状态为 `ready` / 待确认 = 0 / `是否允许正式生成：是`

- 路径：`03_Output/PRD_{功能名}_v{版本}_{日期}.md`
- 默认不覆盖已有文件
- 生成后更新 `_WORK_SHEET.md` 状态为 `completed`

---

## 生成流程

### 第一步：读取状态源

从 `_WORK_SHEET.md` 提取：功能名称、当前状态、待确认数、是否允许正式生成。

### 第二步：判定输出模式

1. 用户明确要求"草稿" → 草稿模式
2. 用户要求正式 PRD 但条件不足 → 拒绝
3. 条件满足 → 正式生成

### 第三步：装配文档内容

内容来源优先级：
1. `_WORK_SHEET.md`
2. `01_Input/Context.md`
3. `@代码文件` 扫描结果
4. `docs/templates/PRD_Template.md` 的章节结构

模板是结构约束，不是内容发明器。不得因模板存在而补写原始输入中没有的信息。正式 PRD 中严禁保留 Q&A、待确认列表等过程信息（草稿除外）。

### 第四步：生成文件

- 正式：`03_Output/PRD_{功能名}_v{版本}_{日期}.md`
- 草稿：`03_Output/PRD_{功能名}_v{版本}_{日期}_draft.md`
- 已存在则追加序号

### 第五步：回写 `_WORK_SHEET.md`

更新状态、阶段、文件路径、输出记录。

---

## "现状逻辑"章节规则

- **有 `@代码文件`**：基于实际代码提取，标注 `(代码位置: 文件名:行号)`
- **无 `@代码文件`**：保留章节，写 `> 待扫描：本次未提供 @代码文件，暂不补充现状逻辑细节。`

---

## 正式 PRD 最小章节集

- 业务背景、需求目标、适用范围、现状逻辑、功能详细设计（业务规则、交互规则）、变更对比、限制规则、异常处理、遗留事项、变更记录

---

## 一句话行为准则

"先读工作台，再决定拒绝、草稿或正式生成；待确认未清零时，正式 PRD 一律不放行。"
