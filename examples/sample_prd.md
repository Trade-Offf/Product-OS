## 1. 背景与目标

### 1.1 业务背景

当前平台用户分为普通用户和 VIP 用户，VIP 通过充值获得，仅有单一级别。用户充值一次后缺乏持续成长激励，留存和复购效果有限。（原文："目前平台用户分为普通用户和VIP用户（充值就是VIP），但VIP只有一个级别，没有成长感，用户充完一次就不管了。"）

### 1.2 核心目标

搭建青铜、白银、黄金、钻石四级会员等级体系，以积分驱动升级，通过差异化权益提升用户留存和付费转化。（原文："我们打算做一个会员等级体系，主要是想提升用户留存和付费转化。"）

## 3. 功能详细设计

### 3.1 功能概述

基于积分体系的四级会员成长路径，覆盖 App、Web、管理后台三端，支持自动升级、等级展示、权益发放及后台管理。

### 3.2 等级体系定义

| 等级 | 积分阈值 | 权益 |
| ---- | -------- | ---- |
| 青铜 | 0 | 无特殊权益（原文："青铜：无特殊权益，就是个起点"） |
| 白银 | 500 | 95 折优惠（原文："享受 95 折优惠"） |
| 黄金 | 2,000 | 9 折优惠 + 专属客服（原文："享受 9 折优惠 + 专属客服"） |
| 钻石 | 5,000 | 85 折优惠 + 专属客服 + 生日礼包（原文："享受 85 折优惠 + 专属客服 + 生日礼包"） |

### 3.3 积分规则

| 积分来源 | 规则 |
| -------- | ---- |
| 消费金额 | 每消费 1 元 = 1 积分（原文："消费 1 元 = 1 积分"） |
| 活跃天数 | 每活跃 1 天 = 5 积分（原文："活跃 1 天 = 5 积分"） |

- 升级为自动触发：用户积分达到对应阈值时立即升级。（原文："升级是自动的，达到积分就升"）
- 积分有效期：本期不做。（原文："积分有效期先不做，后面再说"）
- 降级规则：本期不纳入。（原文："降级的话还没想好，可能按季度评估，也可能不降，先讨论下"）

### 3.4 功能模块拆解

#### 3.4.1 App 端 -「我的」页面

**功能说明**：
用户可在「我的」页面查看当前会员等级和升级进度条。（原文："需要在 App 的「我的」页面展示当前等级和进度条"）

**展示规则**：
- 展示当前等级名称及图标（图标待设计交付）（原文："等级图标让设计出，还没出"）
- 展示当前积分和升级进度条

#### 3.4.2 Web 端 - 个人中心

**功能说明**：
与 App 端展示内容同步。（原文："Web 端的个人中心也要同步展示"）

#### 3.4.3 管理后台 - 用户等级管理

**功能说明**：
运营人员可查看全平台用户等级分布情况，并支持手动调整个别用户等级。

**功能清单**：
- 查看用户等级分布（各等级人数）。（原文："管理后台需要能查看用户等级分布"）
- 手动调整个别用户等级。（原文："可以手动调整个别用户的等级（比如大客户）"）

> ❓ 遗留：手动调级时是否需要填写原因、积分如何同步处理，待确认后补充。

#### 3.4.4 API/后端

**接口清单**：

| 接口 | 用途 | 调用方 |
| ---- | ---- | ------ |
| 等级查询接口 | 返回用户当前等级、积分信息 | App / Web 前端（原文："后端要提供等级查询接口给前端调用"） |
| 积分变动记录接口 | 返回用户积分变动明细 | App / Web 前端（原文："还有一个积分变动的记录接口"） |
| 历史积分回填（一次性） | 上线时根据历史数据写入用户积分 | 后端脚本（原文："老用户上线时需要做一次历史积分回填"） |

## 4. 变更对比

| 维度 | 现状 | 改版后 | 变更原因 |
| ---- | ---- | ------ | -------- |
| 会员模型 | 普通 / VIP 两级 | 青铜 / 白银 / 黄金 / 钻石四级 | 单一 VIP 缺乏成长感 |
| 升级方式 | 充值即 VIP | 积分达标自动升级 | 激励持续消费和活跃 |
| 权益 | VIP 统一权益 | 按等级差异化权益 | 差异化提升付费意愿 |
| 管理能力 | 无等级管理 | 后台可查看分布、手动调级 | 运营需要灵活管理大客户 |
