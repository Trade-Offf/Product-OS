# Product-OS

<p align="center">
  <img src="./Logo.png" alt="Product-OS Logo" width="320" />
</p>

双源驱动的 PRD 智能生成系统，把零散需求整理成结构化工作台，再结合历史背景与真实代码逻辑，生成可直接交付的产品文档。

> 优质 PRD = (历史背景 + 代码现状 + 新需求) × 标准化模版
>
> `Context.md` + `@代码文件` + `01_Input/*` × `docs/templates/PRD_Template.md`

---

## 产品价值

`Product-OS` 用来解决 PRD 产出中的三个高频问题：

- 需求输入分散，会议纪要、草稿、PDF 很难直接变成正式文档
- AI 缺少历史背景时，容易遗漏上下文或误判决策原因
- 需求涉及已有功能时，如果不结合代码现状，PRD 容易偏离真实系统

它把需求处理统一为一条主线：

`输入 → 整理 → 输出`

- 输入：把原始需求和背景材料放进 `01_Input/`
- 整理：生成 `_WORK_SHEET.md`，集中补齐待确认问题
- 输出：结合 `Context.md`、`@代码文件` 和模版生成正式 PRD

---

## 核心原理

### 核心公式

```text
优质 PRD = (历史背景 + 代码现状 + 新需求) × 标准化模版
         = Context.md + @代码文件 + 01_Input/* × PRD_Template.md
```

### 关键对象

| 对象                             | 作用         | 说明                                             |
| -------------------------------- | ------------ | ------------------------------------------------ |
| `01_Input/Context.md`            | 注入历史背景 | 来自 Lark 或其他资料的历史沿革、旧决策、遗留问题 |
| `01_Input/*`                     | 承接原始需求 | 支持 Markdown、PDF、文本等原始材料               |
| `_WORK_SHEET.md`                 | 承接整理结果 | 是需求工作台，不是最终交付物                     |
| `@代码文件`                      | 注入现状逻辑 | 用于提取真实代码中的状态、参数、规则和边界       |
| `docs/templates/PRD_Template.md` | 统一输出结构 | 约束最终 PRD 的章节和表达方式                    |

### 工作原则

- AI 负责整理，不负责发明需求
- 输入区允许随意表达，整理阶段只提取事实
- 逻辑缺口会标记为 `待确认`，不会自动补全
- 修改已有功能时，应尽量结合 `@代码文件` 生成 PRD

## 30 秒快速开始

### 1. 先把 `Product-OS` 和业务代码放进同一个 Cursor 工作区

操作方式：

1. 用 Cursor 打开你的业务代码仓库
2. 将 `Product-OS` 文件夹拖入左侧文件树空白处
3. 选择 `Add Folder to Workspace`

完成后应看到类似结构：

```text
业务代码/
  └── src/...

Product-OS/
  ├── 01_Input/
  ├── 03_Output/
  └── README.md
```

### 2. 为什么多根工作区是必要的

这不是可选优化，而是双源驱动成立的前提。

Cursor 只能读取当前工作区内的文件。`Product-OS` 负责管理需求输入、工作台和 PRD 输出；业务代码仓库负责提供真实的现状逻辑。两者如果不在同一个工作区，会直接影响下面两件事：

- 无法稳定使用 `@` 引用业务代码文件
- AI 无法同时读取“需求材料”和“代码事实”

而 `Product-OS` 的核心能力正是把这两部分合并使用：

- 从 `01_Input/` 和 `_WORK_SHEET.md` 读取需求语义
- 从 `@代码文件` 提取现状逻辑

所以，多根工作区不是使用建议，而是主流程能够跑通的基础条件。

### 3. 放入需求并执行第一条指令

把原始材料放入 `01_Input/` 后，在 Cursor 中输入：

```text
整理一下
```

如果需求涉及已有功能，在生成 PRD 时补一句：

```text
生成文档，扫描 @业务代码/src/.../某个文件.java 的相关逻辑
```

## 完整使用流程

### 流程总览

```text
Step 0  准备背景      Lark / 历史资料 -> 01_Input/Context.md
Step 1  录入需求      原始需求文件 -> 01_Input/
Step 2  整理需求      整理一下 -> _WORK_SHEET.md
Step 3  补齐问题      填写待确认项 -> 更新 _WORK_SHEET.md
Step 4  生成 PRD      生成文档 + @代码文件 -> 03_Output/PRD_*.md
```

### Step 0：准备背景

把本次需求相关的历史资料粘贴到 `01_Input/Context.md`。

建议内容：

- 历史版本沿革
- 之前的产品决策
- 已知限制
- 遗留问题
- 用户反馈

`Context.md` 的作用不是写当前需求，而是告诉 AI：**这个需求所处的历史背景是什么**。

格式没有严格要求，随意粘贴即可。

### Step 1：录入新需求

把本次需求相关的原始材料放入 `01_Input/`。

支持输入：

- Markdown
- PDF
- 纯文本
- 会议记录
- 需求草稿
- 脑图导出
- 截图描述

这里不要求结构化，重点是保留原始信息。

### Step 2：整理需求

在 Cursor 中输入：

```text
整理一下
```

AI 会读取 `01_Input/` 中的材料，并生成根目录下的 `_WORK_SHEET.md`。

这个文件是整个流程中的**需求工作台**，通常包含：

- `Part 0`：业务背景摘要
- `Part 1`：待确认问题
- `Part 2`：详细拆解报告
- `Part 3`：代码扫描建议

文件顶部还会附带状态信息，例如待回答问题数量和当前处理状态。

### Step 3：补齐待确认问题

`_WORK_SHEET.md` 不是过程残留，而是生成正式 PRD 前的确认层。

你需要做的事情：

1. 阅读 `Part 1` 中的待确认问题
2. 在 `**[回答]**:` 后填写答案
3. 将状态从 `[⏳ 待回答]` 改为 `[✅ 已回答]`

这样做的目的，是把原始材料中的含混描述先转为明确决策，再进入正式文档阶段。

### Step 4：生成 PRD

在 Cursor 中输入：

```text
生成文档
```

如果是修改已有功能，建议附带代码引用：

```text
生成文档，扫描 @业务代码/src/main/java/GridStrategy.java 的止盈逻辑
```

生成阶段会组合以下信息：

- `_WORK_SHEET.md` 中已确认的需求拆解
- `01_Input/Context.md` 中的历史背景
- `@代码文件` 中的现状逻辑
- `docs/templates/PRD_Template.md` 中的标准结构

输出文件：

```text
03_Output/PRD_{功能名}_v{版本}_{日期}.md
```

如果未提供 `@代码文件`，PRD 中的“现状逻辑”章节会标记为待扫描，而不会凭空补写。

## 输入与输出说明

### 输入

| 位置                                | 内容                       | 作用                         |
| ----------------------------------- | -------------------------- | ---------------------------- |
| `01_Input/Context.md`               | 历史背景、旧决策、遗留问题 | 补足业务上下文               |
| `01_Input/*.md` / `*.pdf` / `*.txt` | 新需求、会议记录、参考材料 | 提供原始需求输入             |
| `@代码文件`                         | 相关实现代码               | 提供现状逻辑、参数和边界条件 |

### 中间产物

| 文件             | 作用                                             |
| ---------------- | ------------------------------------------------ |
| `_WORK_SHEET.md` | 需求工作台，承接整理结果、问题确认和代码扫描建议 |

### 输出

| 位置                       | 内容       |
| -------------------------- | ---------- |
| `03_Output/PRD_*.md`       | 完整 PRD   |
| `03_Output/PRD_*_light.md` | 精简版 PRD |

## 示例演示

下面是一个完整但最小化的闭环示例。

### 1. 准备背景

`01_Input/Context.md`

```markdown
# 网格交易历史背景

## 历史版本

- v1.0：上线基础网格交易
- v1.5：增加止盈功能，触发后直接下市价单

## 用户反馈

- 市价单止盈滑点较大，希望支持限价单
```

### 2. 录入需求

`01_Input/止盈优化需求.md`

```markdown
# 网格止盈优化

## 需求

1. 止盈支持限价单
2. 用户可设置价格偏移比例（0.1% - 5%）
3. 限价单超时未成交，自动转市价单
```

### 3. 整理并补齐问题

在 Cursor 中输入：

```text
整理一下
```

AI 生成 `_WORK_SHEET.md`，其中会包含待确认问题，例如：

```markdown
### Q1 [⏳ 待回答]

**原文**: "限价单超时未成交，自动转市价单"
**问题**: 超时时间是多少？
**[回答]**:
```

补充完成后：

```markdown
### Q1 [✅ 已回答]

**原文**: "限价单超时未成交，自动转市价单"
**问题**: 超时时间是多少？
**[回答]**: 30秒
```

### 4. 生成正式 PRD

在 Cursor 中输入：

```text
生成文档，扫描 @业务代码/src/strategy/GridStrategy.java 的止盈逻辑
```

最终产出：

```text
03_Output/PRD_grid_stop_profit_v1.0_20260310.md
```

这份 PRD 会同时包含：

- 来自 `Context.md` 的业务背景
- 来自代码扫描的现状逻辑
- 来自需求输入的目标变更
- 基于标准模版组织后的正式章节

## 常见问题

### 为什么一定要用多根工作区？

因为主流程需要同时读取 `Product-OS` 内的需求材料和业务仓库中的代码文件。两者不在同一个工作区时，`@代码文件` 无法稳定工作，双源驱动也就失效。

### `Context.md` 必须填写吗？

不是必须，但建议填写。没有它时，PRD 仍可生成，只是“业务背景”会缺少历史上下文。

### `@代码文件` 必须提供吗？

不是必须。全新功能通常可以不提供；修改已有功能时，建议提供相关代码文件，否则“现状逻辑”章节会标记为待扫描。

### `_WORK_SHEET.md` 是什么？

它是从原始输入到正式 PRD 之间的确认层。它负责收敛背景、暴露待确认问题、整理规则和代码扫描建议，不是一次性草稿。

### 可以同时处理多个需求吗？

建议一次只处理一个需求。这样 `Context.md`、`_WORK_SHEET.md` 和最终 PRD 的对应关系最清晰。

### PRD 模版可以自定义吗？

可以。直接编辑 `docs/templates/PRD_Template.md` 即可。

## 扩展能力

除主流程外，项目还支持按需扩展的技能能力。这些能力不影响 PRD 主链路，但可以在后续协作中继续使用。

| 技能                     | 用途                                         |
| ------------------------ | -------------------------------------------- |
| `Frontend_Design`        | 生成前端页面、组件、落地页、仪表盘等界面代码 |
| `Mermaid_Expert`         | 生成流程图、时序图、ER 图等 Mermaid 图表     |
| `Wireframe_To_Prototype` | 将线框图、字段布局、ASCII 草图转成高保真原型 |

### 常见触发词

| 触发词                                         | 对应能力                 |
| ---------------------------------------------- | ------------------------ |
| `做页面` / `做组件` / `美化`                   | `Frontend_Design`        |
| `画流程图` / `画 Mermaid` / `画时序图`         | `Mermaid_Expert`         |
| `转成原型图` / `高保真原型` / `根据线框做页面` | `Wireframe_To_Prototype` |

### 线框图转原型示例

如果你有一段低保真线框内容，可以直接对 AI 说：

```text
把这段线框图转成高保真原型，不新增功能，只做视觉还原。
```

然后粘贴类似内容：

```text
┌──────────────────────────────────────────────┐
│ 新增带单员 / 编辑带单员                 [X]   │
├──────────────────────────────────────────────┤
│ UID：            [____________________]      │
│ 带单员昵称：      [____________________]      │
│ 头像：            [上传图片] [预览]           │
│ 联系方式：        [____________________]      │
├──────────────────────────────────────────────┤
│                          [取消] [保存]       │
└──────────────────────────────────────────────┘
```

系统会按原始结构生成可预览原型，不新增业务字段，只补充视觉层。

## 工具命令

| 操作       | 命令                           |
| ---------- | ------------------------------ |
| 清除工作区 | `./scripts/clean_workspace.sh` |

### 清除范围

执行 `./scripts/clean_workspace.sh` 后，会删除以下内容：

- `01_Input/` 中的工作文件
- `02_Process/` 中的中间文件
- `03_Output/` 中的输出文件
- `_WORK_SHEET.md`

适合在一个需求处理完成后，清理现场并开始下一轮工作。
